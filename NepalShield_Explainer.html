Step 1 — The Problem

What problem are we solving?

Read this story. This is the exact scenario from the hackathon problem statement.

🚨 Real Fraud Scenario

Sita's bank account shows a transfer of NRs 85,000 to an unfamiliar eSewa merchant at 2:14 AM — very unusual for her. She normally only makes small grocery payments during the day.

At the same time, the bank's system detects that the login came from a new device registered in Dharan — but Sita's phone is in Kathmandu.

A human would immediately say: "This is fraud. Block it."
But current automated systems either miss it or take minutes to respond — by then, the money is gone.

1
🚫
False Positives
Systems block real customers too often. That makes people angry and switch banks.
2
⚖️
Context-Blind Rules
Sending Rs 200 at a grocery store vs Rs 85,000 to unknown person at 2AM is not the same risk — but current systems treat them equally.
3
👤
New User Blindspot
New customers have no history. ML models can't analyze what they don't know — so new accounts are vulnerable.
4
📵
SIM-Swap Attacks
Fraudsters steal someone's phone number. Now they can receive the SMS OTP and bypass verification entirely.

The 5-Agent Pipeline

Every transaction goes through this flow — from detection to verdict — in under 800ms.

⚡ Agents 2, 3, and 4 run in parallel — not one after another. This is how we achieve under 800ms total.

⚡
Velocity Agent
Transaction Frequency
Watches how many transactions happen per minute/hour. If someone suddenly makes 10 transfers in 2 minutes, that's suspicious.
Tracks 1min, 5min, 1hr, 24hr windows
Uses Redis cache for sub-5ms speed
Adapts thresholds per account type
🌍
Geo Agent
Location Anomaly
Checks if the transaction location makes physical sense. If your phone is in Kathmandu but the transaction is from Dharan — that's impossible.
Detects impossible travel scenarios
Flags new/unknown devices
Detects VPN / location spoofing
🧠
Behavior Agent
User Pattern ML
Learns what's "normal" for each user — time of day, transaction size, merchants, channels. Flags anything that breaks their personal pattern.
LSTM neural network per user
Isolation Forest anomaly detection
Peer-group model for new users
⚖️
Synthesis Agent
The Brain / Orchestrator
Collects scores from all 3 agents. Adjusts the weight of each signal based on the transaction type. Issues the final verdict.
Context-aware weighted voting
Remittance vs QR vs ATM logic
Score < 0.35 = PASS | > 0.65 = BLOCK
🔐
OTP Interlock
Dual-Path Verification
When score is in the middle (0.35–0.65), it sends OTP via BOTH email AND SMS. Both must be confirmed. One channel being hacked is not enough.
Email + SMS simultaneously (Sparrow)
SIM-swap detector — checks if SIM changed in 48hrs
90 sec timeout → auto-block

The 3 Possible Verdicts

✅ Pass
0–0.35
Transaction goes through instantly. No friction for the customer.
⚠️ OTP Required
0.35–0.65
Suspicious but not certain. Customer verifies via email + SMS.
🚫 Block
0.65+
High confidence of fraud. Transaction blocked. Fraud team alerted.
Step 4 — The 4 Problems They Want Solved

How we solve each hackathon challenge

The hackathon listed 4 specific problems. Here's exactly how NepalShield solves each one.

1
Minimizing false positives — blocking real customers = they leave the bank
↓ Our Solution
💡
3-tier response: PASS / OTP / BLOCK — not just block or allow. Real customers get OTP instead of hard block. We also train on Nepal-specific data, not western models.
2
Synthesis agent must adapt weights based on context (remittance vs POS vs QR)
↓ Our Solution
💡
Dynamic weight table in the Synthesis Agent. For high-value P2P transfers, graph analysis weighs 30%. For QR payments, velocity weighs 35%. Context changes everything.
3
Cold-start problem: new users with no behavioral history
↓ Our Solution
💡
Peer-group model. New users are matched to similar users (same age, account type, onboarding branch). They inherit that group's baseline until they build personal history (30+ transactions).
4
SIM-swap attacks that compromise both SMS channels simultaneously
↓ Our Solution
💡
Dual-path OTP (email + SMS both required). Plus a telecom API check — if SIM was changed in 48 hours, SMS is disabled entirely and only email is used.
